## Fixed architecture decision

Do not spend time reconsidering the storage architecture.

The n8n Data Table documentation does not establish a transactional unique-key or compare-and-set guarantee sufficient for a concurrent send lock. Therefore:

- Use n8n Data Tables only for ordinary validation records where atomic ownership is not required.
- Reuse the already verified V5 Layer 2 atomic file-lock/state logic for send ownership.
- Package that logic as a tiny internal Docker sidecar named `hmz-send-state`.
- The service must be reachable only on the internal Docker Compose network.
- Do not publish its port to the Windows host.
- n8n must address it as `http://hmz-send-state:5681`.
- The state directory must use a named Docker volume.
- No credentials or prospect message bodies may be stored by the sidecar.

This is a Validation-MVP mechanism, not a production-scale database claim.

## Sidecar contract

Implement a minimal HTTP JSON service using Node.js built-ins.

Required endpoints:

- `GET /health`
- `POST /v1/send/acquire`
- `POST /v1/send/transition`
- `GET /v1/send/:sendKey`
- `POST /v1/error`
- `GET /v1/error/:errorId`

Requirements:

- Bind to `0.0.0.0:5681` inside the container.
- No published host port in Docker Compose.
- Stable send key derived from canonical inbound Email ID, sender, recipient, and policy/template identity.
- Random body marker must not be part of the send key.
- Exclusive atomic acquisition.
- Durable state survives process/container restart through the named volume.
- A concurrent second acquisition returns a blocked result.
- A later sequential rerun returns a durable-state-exists result.
- Forward-only states:
  - `READY`
  - `LOCKED`
  - `DRY_RUN_OK`
  - `SUBMITTING`
  - `SENT`
  - `SEND_UNCERTAIN`
  - `SENT_RECONCILED`
  - `HUMAN_REVIEW_ZERO_MATCHES`
  - `HUMAN_REVIEW_MULTIPLE_MATCHES`
  - `PERMANENT_FAILURE`
  - `AUTH_OR_PLAN_FAILURE`
  - `INVALID_REPLY_TARGET`
  - `RETRY_EXHAUSTED`
  - `BLOCKED`
- No transition from an uncertain or human-review state back to submission.
- Store only sanitised identifiers, hashes, state, timestamps, attempt counts, HTTP class, and operator action.
- Never store API keys, Authorization headers, full reply bodies, full webhook payloads, or raw API responses.
- Error records must be sanitised and written atomically.

Copy and adapt the verified V5 logic rather than rewriting atomic locking from scratch.

## Sender workflow

Generate an import-ready inactive n8n workflow JSON.

Use only standard built-in nodes and patterns already present in the existing exports where possible.

Required high-level flow:

1. Execute Workflow Trigger.
2. Validate Decision Engine output and required identifiers.
3. Re-run configuration, approval, campaign, sender, draft, template-variable, suppression, and safety gates.
4. Compute stable send key.
5. Acquire ownership through `hmz-send-state`.
6. Route blocked duplicate/rerun to a terminal no-send result.
7. Execute required suppression actions through a mock adapter only.
8. Verify suppression results.
9. Block unsubscribe confirmation until both source-campaign action and exact workspace email block-list action are verified.
10. Apply DRY_RUN gate.
11. In DRY_RUN, transition to `DRY_RUN_OK` and return a sanitised result.
12. Include an unreachable validation-only live-adapter contract for the verified Instantly fields, but do not bind credentials and do not make a live HTTP request.
13. Implement the V5 error classification, bounded retry policy, `Retry-After`, and uncertain-send state as deterministic code/contracts.
14. Implement read-only reconciliation contract with zero/one/multiple outcomes.
15. Never issue a second POST after `SEND_UNCERTAIN`.
16. Return one structured terminal object.

Hardcoded validation settings:

- `OPERATING_MODE=VALIDATION`
- `DRY_RUN=true`
- `LIVE_CAMPAIGNS=[]`
- `LIVE_CREDENTIAL_READY=false`

No path may override these through input.
