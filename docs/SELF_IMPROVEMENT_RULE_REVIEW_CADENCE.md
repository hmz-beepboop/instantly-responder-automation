# Self-Improvement Rule Review Cadence

**Version:** 1.0  
**Date:** 2026-06-24

---

## Purpose

This document defines when and how the owner reviews rule candidates generated by the self-improving layer.

---

## Review Triggers

### Trigger 1 — After each review session (event-driven)

Every time the owner reviews and approves/edits cases in the HumanApproval review form, new learning events and rule candidates may be generated. After any session involving corrections, run:

```powershell
.\scripts\SL-PHASE-3-rule-candidate-review-console.ps1 -ListCandidates
```

If there are new `proposed_shadow` candidates, review them before the next campaign send wave.

### Trigger 2 — Weekly cadence (calendar-driven)

Review all pending rule candidates at least once per week regardless of session activity. This ensures candidates do not accumulate stale.

**Recommended day:** Monday morning before the week's campaign activity.

### Trigger 3 — After observing a pattern (quality-driven)

If the owner notices the same type of draft error occurring multiple times across different prospects, that pattern is a strong signal to:
1. Check if a correction has been captured
2. Create a manual rule candidate if not already captured
3. Approve the candidate promptly

---

## What to Look for in Each Candidate

| Field | What to assess |
|-------|---------------|
| `proposed_rule_text` | Is this rule clear, specific, and generalisable? |
| `trigger_condition` | Will this trigger on the right cases? Too broad? Too narrow? |
| `correction_reason` | Does the correction reason justify creating this rule? |
| `source_case_id` | Was this a genuinely bad output that warrants a rule? |
| `confidence` | High confidence corrections deserve more weight |
| `category` | Does this apply to a category we regularly handle? |

---

## Approve / Reject / Amend Criteria

### Approve when:
- The rule is specific and actionable
- The trigger condition is clear and not too broad
- The correction was genuinely necessary (not reviewer preference)
- The rule would not override safety gates
- The rule applies to a pattern likely to recur

### Reject when:
- The case was a one-off that won't recur
- The rule is too vague to be useful
- The rule conflicts with approved reply policy
- The correction was reviewer preference, not a real error
- The rule touches pricing, compliance, legal, or GDPR (these belong in approved reply policy, not auto-rules)

### Amend before approving when:
- The rule text is close but needs tightening
- The trigger condition is too broad and needs narrowing
- The correction reason is valid but the proposed rule text is not the best expression of it

## How Human Edits Continue Producing Proposed_Shadow Rules

After Phase 4, this is automatic. The flow is:

1. Reviewer submits a correction → sl_p2_correction_event created
2. Proxy writes the event to hmz-learning-events
3. A proposed_shadow rule candidate is written to hmz-rule-candidates
4. Candidate appears in the review console with status = proposed_shadow
5. Owner runs the review console to inspect and act

No manual intervention is needed to start the capture. It is always active for cases that go through the HumanApproval workflow.

## Deprecating or Rolling Back Active Rules

### Deprecate when:
- A rule has been active for 30+ days and the improvement is no longer relevant
- The business context has changed (e.g. offer maturity changed, pricing became fixed)
- A better rule supersedes it

### Roll back immediately when:
- An active rule produced a demonstrably harmful output
- A rule is triggering on cases it should not
- An autonomous audit (when live) surfaces repeated bad outputs from a rule

### Rollback command:
```powershell
.\scripts\SL-PHASE-4I-phase3-proxy-write-repair.ps1 -RollBack -RuleId RC-XXX
```

Then verify the rule is removed from Decision node D:
```powershell
.\scripts\SL-PHASE-4J-self-improvement-verification-suite.ps1 -RunS7
```

## Rule Lifecycle Summary

```
proposed_shadow → owner review → approved OR rejected OR amended
approved → injected into Decision node D (ACTIVE_RULE_GUIDANCE)
active → monitored → deprecated OR rolled_back if harmful
```
