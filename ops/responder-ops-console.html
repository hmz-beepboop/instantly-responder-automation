    Start Supervised Responder — Guidance

    This console cannot start anything. Starting is a manual owner action in the n8n UI, done only after every step below. "Supervised" means every reply still goes through human approval.

    1. Read the latest entry in OPERATION_HANDOFF.md — it is the execution truth. Do not start from README or older docs.
    2. Run the production target guard: pwsh -File ./scripts/assert-hmz-production-target.ps1. Only https://n8n.hmzaiautomation.com is valid.
    3. Verify workflow versions in the n8n UI match the handoff's recorded versionIds (Decision, HumanApproval, Sender). A mismatch means someone changed production — stop and investigate.
    4. Run the harness: python scripts/SL-PHASE-5Q-self-improvement-behavioural-closure.py — must be fully green.
    5. Run the secret scan: python scripts/scan-workflow-exports-for-secrets.py.
    6. Confirm the campaign's CRR (docs/campaign-readiness/) is complete and signed. No CRR → no launch.
    7. Confirm expected workflow states in the n8n UI: Intake/Decision/HumanApproval/Sender active; Shadow Evaluator inactive.
    8. Send one probe reply from the owned test lead and walk it through review before exposing any real prospect traffic.
    9. Confirm Google Chat notifications arrive and the review link opens.
    10. Record the start (date, versions, operator) in OPERATION_HANDOFF.md.

    Stop Responder — Guidance

    This console cannot stop anything. Stopping is a manual owner action in the n8n UI.

    1. To stop NEW replies being processed: deactivate the Intake/monitor workflow in the n8n UI. In-flight review cases can still be finished by reviewers.
    2. Do NOT deactivate the Sender workflow while a send is in flight; let executions finish first (check the Executions list is idle).
    3. To stop everything including approvals: additionally deactivate HumanApproval — outstanding review links will stop working; note them first.
    4. Never "stop" by deleting workflows, credentials, or DataTable rows. Deactivate only.
    5. If stopping because of a suspected fault (duplicate, blank body, wrong sender): stop FIRST, collect evidence SECOND (see Diagnose Issue), change nothing else, and record the fault in docs/INSTANTLY_RESPONDER_FAULT_LEDGER_AND_SCALE_READINESS.md.
    6. Record the stop (reason, time, what was deactivated) in OPERATION_HANDOFF.md.
    7. Restart only via the full Start checklist — never by just toggling workflows back on after a fault.

    Diagnose Issue

    Pick the symptom. Follow the stop rule before anything else. This tool only explains — it cannot inspect production.
